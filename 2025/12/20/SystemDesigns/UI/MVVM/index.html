Model View ViewModel (MVVM)

发表于 2025-12-20 分类于 SystemDesigns ， UI

核心总结

MVVM是MVC架构的进阶优化方案，核心目标是实现UI层（View）与业务逻辑层的彻底解耦：View层仅负责可视化展示和用户输入转发，所有业务逻辑、数据管理由ViewModel层承接；通过FieldNotify特性实现View与ViewModel的数据自动双向同步，无需手动编写数据更新逻辑，需开启UE的“UMG ViewModel”插件才能使用该架构。

MVVM架构（MVC的进阶优化）

MVC架构虽实现了View与Controller的解耦，但View层（Widget）的按钮回调仍需间接绑定游戏逻辑（通过Controller执行）；MVVM则进一步强化解耦，核心设计原则：

  • View层（Widget）：仅做两件事——调用ViewModel中定义的按键回调函数（无任何游戏逻辑）；将UI组件（如TextBlock）与ViewModel的变量绑定，负责可视化展示；
  • ViewModel层：承接所有业务逻辑（如数据修改、关卡跳转），管理UI所需的所有数据，通过FieldNotify实现数据变更后View层自动更新；
  • 核心优势：FieldNotify特性让View层无需编写“从Controller读取数据”的逻辑，组件与ViewModel变量绑定后，变量值变更会自动同步到UI显示，大幅减少重复代码。

使用MVVM架构需先在UE编辑器中启用“UMG ViewModel”插件。

自定义ViewModel类需继承UMVVMViewModelBase，5.3及以上版本需注意：Widget与ViewModel的绑定需满足“UI子组件与ViewModel变量关联”，否则绑定无法生效。核心步骤如下：

在Widget的Designer界面，打开“ViewModels”面板，添加自定义的ViewModel类，完成ViewModel与Widget的关联。

宏说明

  • UE_MVVM_SET_PROPERTY_VALUE：MVVM框架提供的宏，赋值同时触发通知，需在Setter中调用。

3. 绑定UI组件与ViewModel变量

在Widget的Designer界面打开“ViewBindings”面板，选择目标UI子组件（如TextBlock），关联到ViewModel中对应的FieldNotify变量，完成数据绑定。

特殊场景处理

若Widget无需要显示的变量（仅需调用ViewModel逻辑），可临时绑定一个调试变量，并将该组件的Opacity设为0，确保ViewModel绑定生效。

ViewModel的实例化（支持多控件复用）

ViewModel并非严格绑定单个Widget，支持多个不同Widget共享同一个ViewModel实例（如加载界面的“Play”“Delete”“Select Slot”按钮），确保数据一致性：

  • 数据层面：所有控件访问的是同一套数据（如选中的存档档位）；
  • 逻辑层面：不同控件的回调函数（如PlayButtonPressed、SelectSlotPressed）均封装在同一个ViewModel中。

Widget Switcher（界面控件切换）

Widget Switcher用于管理同界面内的控件切换（如加载界面的“新建存档”“选择存档”面板），虽不属于MVVM核心，但常与MVVM配合使用：

  • 布局层面：在Overlay上放置Widget Switcher，内部添加多个待切换的Widget，通过索引标识不同面板；
  • 切换逻辑：调用SetActiveWidgetIndex函数并传入目标索引，即可切换显示的Widget。

MVVM设计原则

编写ViewModel代码时需遵循“忘掉View层”的原则：

  • 所有业务逻辑、数据管理、委托交互均在ViewModel层完成，ViewModel之间可直接通信；
  • View层仅作为“可视化外壳”，仅负责展示数据和转发用户输入，不参与任何逻辑决策；
  • 最终实现“ViewModel承载所有核心逻辑，View仅为附属展示层”，最大化解耦效果。
