Head Up Display (HUD)

发表于 2025-12-20 分类于 SystemDesigns ， UI

核心总结

HUD（Head Up Display）是UE UI系统的顶层载体，核心职责为托管全局Overlay Widget（所有子UI控件的父容器）和复用型WidgetController实例，不承载具体业务逻辑；通过统一初始化Overlay Widget并绑定对应的WidgetController，适配MVC架构下UI与游戏逻辑的解耦需求，保证同类型WidgetController的共享复用，减少冗余实例创建。

HUD 的核心定位与功能

HUD并非业务逻辑的执行载体，而是屏幕所有UI控件的“顶层容器”，需在GameMode的配置项中指定对应的HUD Class以生效；
屏幕上的可视化界面仍由Widget编辑实现，若存在多个子UI控件（如血条、技能栏、提示信息），会统一整合到一个Overlay Widget中管理，HUD的核心功能聚焦于Overlay Widget的创建、初始化，以及WidgetController的托管复用。

1. 创建并初始化全局Overlay Widget

HUD负责创建Overlay Widget实例，完成其与对应WidgetController的绑定，并将Overlay添加到视口，子控件可在Overlay的创建流程中按需初始化，无需HUD直接干预：

初始化逻辑说明

  • 传入PC、PS、ASC、AS等核心游戏对象，构建WidgetController所需的参数，确保Controller能对接游戏核心数据；
  • 绑定Overlay Widget与对应的WidgetController后，立即广播初始数据，保证UI加载时能同步最新的游戏状态；
  • Overlay添加到视口后，成为所有子UI控件的父容器，统一管理屏幕UI的层级、显示与隐藏。

2. 托管复用型WidgetController（适配MVC架构）

在MVC架构的UI系统中，每个Widget需绑定专属的WidgetController，但同功能类型的WidgetController需共享复用（如所有属性类子控件共用一个AttributeMenuWidgetController），避免重复实例化造成资源浪费。

Overlay Widget本身也是Widget，需绑定自身的WidgetController，因此不适合作为其他WidgetController的托管载体；HUD作为与游戏会话同生命周期的顶层容器，是存储复用型WidgetController的最优选择，所有Widget获取Controller时均从HUD中读取，无实例时才创建并初始化：

WidgetController复用逻辑

  • 懒加载创建：仅当对应的WidgetController实例不存在时，才通过NewObject创建，并完成参数初始化、依赖回调绑定；
  • 全局复用：HUD的生命周期贯穿游戏全程，存储的WidgetController可被所有子Widget重复调用，保证数据同步的一致性；
  • 解耦保障：Widget只需从HUD获取Controller，无需关心Controller的创建与初始化细节，符合MVC“View层与Controller层解耦”的核心原则。
